If it's COBOL / mainframe

COBOL runs on IBM mainframes (z/OS). It processes transactions in batch jobs — large files of records on a schedule, not continuously. Data lives in flat files or VSAM, not relational databases. There is no native REST API and no Kafka producer. Data gets out via file extracts, MQ message queues, or DB2 CDC if a DB2 layer exists.

Egress optionsScheduled file extract, MQ polling, or DB2 CDC via Debezium — real-time CDC only possible if DB2 is in the picture
Batch rhythmPipeline inherits the batch cadence — you're often getting 50k records at 2am, not 50k TPS in real time

COBOL gotchas — these will impress interviewers

EBCDIC encodingMainframes don't use ASCII/UTF-8. Raw ingest without conversion produces silent garbage. Normalization layer must handle this.
Packed decimalCOMP-3 format — PIC 9(7)V99 COMP-3 is not a string or integer, it's compressed binary. Needs a decoder or you get data quality bugs.
Fixed-width recordsNo delimiters. Field X starts at byte 142, is 8 bytes wide — because the copybook says so. No copybook = no schema.
Batch window riskIf the overnight job fails or runs long, your entire downstream is stale. Design for late data explicitly — don't assume the extract arrives on time.

The one-liner

"COBOL cores typically don't support real-time egress natively, so the integration pattern shifts from CDC to scheduled extract or MQ polling. Key risks are EBCDIC encoding, packed decimal fields, and batch window dependencies — all absorbed in the normalization layer before anything downstream sees the data."

Tier 1 reality — middleware is almost always already there

Most large banks have already wrapped their COBOL core in middleware. You're rarely integrating directly with the file system — you're integrating with whatever the mainframe team has already exposed.

IBM MQMost common. If it's there, lead with a Kafka Connect MQ source connector — mainframe team changes nothing, you get durable ordered messages from day one.
DB2Often the hidden gem. Many COBOL shops write transaction results to DB2 for reporting. If DB2 exists, Debezium has a connector — near real-time CDC without any mainframe-side changes.
CICSThe online transaction layer — handles real-time COBOL transactions. CICS-to-MQ bridges are common. Knowing this word signals genuine familiarity.
ESBMany banks built a custom enterprise service bus in the 2000s. Everyone hates it, nobody has replaced it. Treat it like MQ — tap it, don't fight it.

Tricks of the trade

✓Always land files in object storage first — never process in place. S3/GCS as the landing zone before any parsing begins.
✓Validate the trailer record — COBOL extracts almost always end with a control record containing row counts and totals. Check it before processing a single row. Mismatch = reject the whole file.
✓Version your copybook parser — mainframe teams add fields by appending to the end of the record, making it longer. Your parser needs to handle variable record lengths and you need a layout versioning strategy.
✓Design for late data — batch windows slip. Don't assume the extract arrives on schedule. Your pipeline needs a late-arrival handling path, not just a happy path.
✓Ask for DB2 before anything else — "Does your COBOL system write to DB2 anywhere in the flow?" is the single best question you can ask. If yes, your entire ingestion strategy changes.

The political trick

The mainframe team has their own release cadence, their own change board, and deep skepticism of "the cloud people." Every ask that requires a mainframe-side change will take 6 months minimum. Design your integration to need nothing from them — "we just want to read your MQ queue" beats "can you emit Kafka events" every time.

If MQ is confirmed — say this

"In that case I'd lead with a Kafka Connect MQ source connector — keeps the mainframe team completely out of scope and we get durable, replayable messages from day one."

Exec translation

"Legacy core""We're reading from the system of record without touching it — non-invasive, zero risk to the source."

HA consideration

You don't control source HA — so design for source unavailability. Your ingestion layer must buffer and resume. Never make your pipeline dependent on source uptime.

Key properties to name

DurabilityEvents must survive producer restarts — acks=all, idempotent producer
BufferingIngestion decoupled from downstream — source outage doesn't lose data
ReplayLog-based CDC (Debezium) gives you the database changelog — you can replay from any offset
Non-invasiveDebezium reads transaction log — no triggers, no schema changes on source

HA / 5-nines at this layer

✓Debezium connector in distributed mode — multiple workers, auto-rebalance on failure
✓Kafka producer with acks=all + min.insync.replicas=2 — no silent loss
✓Source outage → connector pauses, resumes from last committed offset on recovery
✓Snapshot catch-up path for cold start or long outage — reconcile from snapshot then resume CDC
Failure story hook: "We had a CDC outage that lasted 18 minutes. Because we'd designed for replay, we resumed from the committed offset and caught up in under 4 minutes with no data loss."

Interview signal

Mentioning log-based CDC vs query-based polling immediately shows depth. Query-based misses deletes and has higher source load. Log-based is the right choice for financial systems.

What to say about Kafka cold

Partition keyAlways account_id for financial — strong per-account ordering, no cross-account dependency
Replication factorRF=3 minimum, min.insync.replicas=2 — single broker loss is invisible to producers
Idempotent producersenable.idempotence=true — prevents duplicates on retry, essential for exactly-once semantics
Log compactionFor state topics (latest value per key). Retention for audit topics (all events). Don't conflate them.
Consumer groupsStatic membership for latency-sensitive paths — avoids unnecessary rebalances on rolling restart

HA / 5-nines Kafka topology

✓3+ brokers across 3 AZs — rack-aware partition assignment so each replica is in a different AZ
✓Controller quorum (KRaft mode) — eliminates ZooKeeper as a separate failure domain
✓Cross-region topic replication via MirrorMaker 2 — active-active multi-region with consumer offset sync
✓Per-partition lag SLI — aggregate lag hides hot partitions. Alert per partition.
Hot partition story: "We had a hot partition because we were keying by transaction type instead of account. One type dominated. We repartitioned by account_id and the lag SLI flattened immediately. Added the per-partition alert so it can't hide again."

Pushback: "Why not Kinesis?"

Kinesis is valid if ops simplicity and AWS-native tooling matter more. Kafka wins on replay semantics, partition control, and ecosystem (Flink, ksqlDB). For a regulated financial system needing deterministic replay, Kafka is the right call.

What canonical schema gives you

Every downstream system speaks one language regardless of source. Schema drift in the source doesn't break consumers — the normalization layer absorbs it.

Schema registryConfluent Schema Registry — Avro/Protobuf schemas versioned and enforced at produce time
Compatibility modeBACKWARD — new consumers can read old messages, old consumers can read new messages
ValidationInvalid events → DLQ immediately, never silently dropped

Canonical ledger event model

transaction_id account_id amount timestamp previous_event_hash source_system
previous_event_hash chains events — deterministic replay from T=0 always produces the same balance. This is your audit guarantee.

HA / 5-nines risk here

Schema registry is a critical dependency. If it's unavailable, producers can't validate schemas and may back-pressure. Run it in cluster mode (3 nodes) and treat it like a control plane component, not an application.

Interview signal

Mentioning schema evolution strategy (backward/forward compatibility) distinguishes you from candidates who treat schemas as a one-time decision.

The two laws

"Idempotent processing — the same event processed twice produces the same result."
"Deterministic logic — given the same input at any time, the output is identical."

Distributed system pillars — must cover all five

PartitioningPer-account via partition key — strong ordering within account, scale across accounts
IdempotencyDedupe store keyed on transaction_hash — exact-once effect even with at-least-once delivery
BackpressureBound consumer concurrency — reject early rather than collapse under burst traffic
Consistency modelStrong within account, eventual across — higher throughput, acceptable trade-off for this use case
Failure handlingDLQ for poison events, circuit breaker on downstream write, retry with jitter

Concurrency patterns

Worker poolsFixed size — unbounded goroutines/threads will OOM under burst
Circuit breakersOpen on error threshold — fast-fail prevents cascade to storage layer
Retry with jitterExponential backoff + random jitter — avoids thundering herd on recovery

AI augmentation — right here, not in the critical path

LLMs belong as a sidecar to processing decisions, not in the hot path. Anomaly detection, RCA assistance, log classification — all async, all advisory.

Strong AI-native workflow
  • LLM generates failure modes → converted to tests
  • LLM critiques design as "Google SRE"
  • LLM produces runbooks from SLOs
  • Human validates via load + property tests
Weak
  • Autocomplete only
  • AI in the write path
  • No human validation gate
  • LLM makes commit decisions
"Intelligence never grants authority." — AI assists, deterministic logic decides.

Financial storage non-negotiables

ImmutabilityAppend-only event store — never update, never delete. Corrections are new events.
UpsertsIdempotent writes keyed on transaction_id — safe to replay without duplicating records
Replay safety"If we replay all events from T=0, do we get the same balance?" — if not, it's not an audit log
CQRSSeparate write model (event store) from read model (materialized balance views) — read performance doesn't compromise write integrity

HA / 5-nines for storage

✓Postgres HA — primary + 2 standbys across AZs, synchronous replication to at least one standby
✓Patroni / pgEdge / RDS Multi-AZ — automated failover, no manual promotion required
✓Point-in-time recovery — WAL shipping to object storage, cross-region replicated. RPO ≈ 0.
✓Connection pooling (PgBouncer) — storms on failover don't exhaust connection limits
✓Multi-region DR — home-region writer, cross-region read replica promoted on failover
5-nines for storage requires the failover to be automatic and tested. "We do a monthly failover drill in staging and measure actual RTO. Last test: 47 seconds."

Active-Active DR — exec translation

TechnicalSingle-writer per account (home region), cross-region replication, reroute on failure
Exec"No downtime if a data center fails — traffic automatically shifts, data is already there."

Design principle

Consumers should subscribe to events or query materialized views — never query the event store directly. The interface layer is read-only against the write path.

Event subscriptionDownstream systems consume from Kafka topics — decoupled, no polling
REST / GraphQLQuery materialized views, not live event log — read performance isolated from write path
Webhooks / alertsPush model for operational events — reconciliation mismatches, lag threshold breaches

Cutover strategy — this layer is your migration gate

1Mirror-only validation — no reads shifted, reconciliation running silently
2Reconciliation parity — confirm 48–72h of hash match before moving traffic
3Shadow posting — writes to both systems, source remains truth
4Gradual read shift — by product line, with instant rollback gate per line
5Write migration — per product line, never big-bang
Exec translation: "We never flip a switch. Each product line migrates independently. If anything looks wrong, we roll back that line in 30 seconds."

5-nines observability stack

MetricsPrometheus with remote write — never lose metrics during the outage you're trying to debug
TracingOpenTelemetry SDK — trace from CDC event through normalization, processing, to write confirmation
AlertingMultiwindow burn rate — 1h + 6h window. A spike looks different than sustained degradation. Only burn rate captures both.
DashboardsGrafana SLO burn rate is the primary on-call view — not raw metrics, not counts
LoggingStructured JSON only — queryable in seconds, not grep'd in minutes

Production incident pattern — model story

DetectionPager alert on SLO burn for write latency
Impact30% of transactions delayed for 6 minutes
ActionTraffic shed → rollback → replay from Kafka
Root causeHot partition due to poor key choice
PreventionRepartitioned by account_id, added per-partition lag SLI + alert
This story shows: metrics-driven detection, structured response, root cause analysis, and prevention. Own every step.

The rule

"Keep AI outside the critical path. Deterministic systems decide. AI assists."

Where AI belongs in this architecture
